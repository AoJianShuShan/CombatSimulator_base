# 单变量敏感性分析设计与落地记录

## 文档目的

这份文档用于冻结“单变量敏感性分析”功能的一期需求、协议口径、前后端结构和实施步骤，避免后续实现过程中出现“前端想要一套、后端返回一套、结果区展示又是另一套”的问题。

## 一期目标

在保持当前战斗模拟器核心规则不变的前提下，新增一个独立的“分析视图”：

- 固定当前整份战斗配置
- 只选择 `1` 个数值变量进行区间扫描
- 每个取值点运行 `N` 场战斗统计
- 横向比较不同取值点下的结果变化趋势

一期只解决最常见、最直接的策划分析场景：

- 指定某个单位
- 指定该单位的某个数值属性
- 配置起始增幅、结束增幅、步进
- 配置每个取值点的模拟场次
- 输出趋势图和明细表

典型例子：

- 红方 `A-1` 的 `攻击` 以当前配置为基准，扫描 `-20 ~ +40`
- 蓝方 `B-2` 的 `护甲` 以当前配置为基准，扫描 `-50 ~ +100`
- 红方 `A-3` 的 `射速` 以当前配置为基准，扫描 `-30 ~ +60`

### 业务范围

- 新增第三个执行视图：`敏感性分析`
- 支持对 `单位属性宏` 中的数值字段做单变量扫描
- 同时支持：
  - 前端本地运行
  - 后端 API 运行
- 结果支持：
  - 页面展示
  - JSON 导出

### 技术范围

- 复用现有 `BattleInput`
- 复用现有 `simulateBattleBatchSummary` 统计逻辑
- 新增分析协议、分析校验和分析执行层
- 不修改现有伤害公式、驱动模式和事件流规则

## 一期非目标

- 不做二维敏感性分析
- 不做多变量联动分析
- 不做技能、Buff、Debuff 等复杂效果分析
- 不保存每个取值点的完整事件日志
- 不为敏感性分析增加新的战斗规则字段
- 不在一期支持“整场战斗参数”扫描，一期只支持“单位属性”扫描
- 不引入第三方图表库

## 为什么单独做一个视图

当前页面已经有：

- `单场模拟`
- `多场统计`

敏感性分析本质上是“同一份配置的另一种执行方式”，不是普通战斗参数，也不是单位详情字段。

如果把敏感性配置硬塞进“整场战斗参数宏”里，会有两个问题：

1. 它不是战斗规则本身，而是分析器自己的配置。
2. 它会污染现有导入、导出、校验和文档口径。

因此更合理的做法是：

- 页面上新增 `敏感性分析` 视图
- 战斗配置继续保持原位不动
- 只在切到敏感性分析时，额外显示一个“敏感性设置”区域

## 前端布局方案

### 顶部视图切换

当前 `单场模拟 / 多场统计` 扩展为：

- `单场模拟`
- `多场统计`
- `敏感性分析`

### 参数区布局

保留现有“整场战斗参数”区域的整体布局和字段位置。

当视图切换到 `敏感性分析` 时，在参数区底部追加一个单独子区域：

- 标题：`敏感性设置`
- 桌面端按 `5` 列紧凑排布，尽量保证高频字段一屏可见
- 字段：
  - `分析目标单位`
  - `分析属性`
  - `当前基准值（只读）`
  - `起始增幅`
  - `结束增幅`
  - `步进`
  - `每点模拟场次`
    默认值为 `100`
  - `扫描点数（只读）`
  - `总模拟次数（只读）`

### 结果区布局

敏感性分析视图下，结果区建议分成 3 块：

1. `分析摘要`
   - 分析单位
   - 当前基准值
   - 增幅范围
   - 结果范围
   - 点数
   - 每点场次
   - 总模拟次数
   - 根种子
2. `趋势图`
   - 第一版使用原生 `SVG`
   - 第一张图展示三条线：
     - 红方胜率
     - 蓝方胜率
     - 平局率
   - 第二张图展示两条线：
     - 红方平均终局净优势
     - 蓝方平均终局净优势
   - 第三张图展示完成时间/回合：
     - 回合制显示平均完成回合
     - 即时制显示平均完成时间
   - 纵轴刻度固定在图左侧，横轴展示增幅范围
   - 图例本身可点击开关，作为对应曲线的显隐控制
   - 只有当前处于开启状态的曲线，才会在图中绘制，也才会参与悬停提示
   - 鼠标移动到某个 `X` 点位时，提示会合并展示该点位下所有“已开启曲线”的数值
   - 如果多条线在同一个点位重叠，不再只显示其中一条，而是并列展示全部开启线的数值
3. `分析明细表`
   - 每行一个取值点
   - 列建议包含：
     - 增幅值
     - 结果值
     - 红方胜率
     - 蓝方胜率
     - 平局率
     - 红方平均终局净优势
     - 蓝方平均终局净优势
     - 平均完成回合或平均完成时间

## 扫描规则

### 一期只支持单位属性扫描

一期 `axis.scope` 固定为 `unitStat`。

也就是说，本期只允许扫描：

- 某个单位的某个数值属性

例如：

- `A-1.attack`
- `B-2.armor`
- `A-3.fireRate`

### 增幅语义

当前版本中，`sweep.start` 和 `sweep.end` 都表示“相对当前基准属性的变化值”，不是绝对属性值。

也就是说：

- 页面上显示的 `当前基准值` 来自当前 draft 中该单位该属性的原始数值
- 实际参与模拟的结果值 = `基准值 + 增幅值`
- 明细表里会同时展示：
  - `增幅值`
  - `结果值`

例如：

- 当前 `A-1.attack = 120`
- 配置 `起始增幅 = -20`
- 则第一个扫描点真正参与战斗的是 `attack = 100`

### 步进规则

- `step` 必须大于 `0`
- `start`、`end`、`step` 都必须满足目标属性自己的步进规则
- `start`、`end` 允许为负数
- 目标属性的最小值约束，继续复用属性宏定义
- 如果 `基准值 + 增幅值 < 属性最小值`，直接报错，不做静默修正

### 方向规则

- 如果 `start <= end`，按升序扫描
- 如果 `start > end`，按降序扫描
- 扫描方向由 `start` 和 `end` 自动决定，用户不需要额外输入正负步长

### 点数计算

设：

- `distance = abs(end - start)`
- `count = distance / step + 1`

要求：

- `distance` 必须能被 `step` 整除
- 否则直接报错，而不是偷偷补最后一个值

这样可以避免展示出来的取值点和用户预期不一致。

## Seed 策略

这是敏感性分析最关键的口径之一。

所有取值点必须共用同一组战斗 seed 序列。

也就是说：

- 第 `1` 个取值点的第 `37` 场
- 第 `2` 个取值点的第 `37` 场
- 第 `3` 个取值点的第 `37` 场

都必须基于同一个派生 seed 运行。

这样横向比较才公平，差异更多来自变量变化，而不是随机波动。

一期直接复用现有：

- `deriveBattleSeed(baseSeed, index)`

## 执行层架构

### 核心原则

敏感性分析不是第三套战斗模拟器，而是“批量统计的上层调度器”。

正确结构应该是：

- `simulateBattle`
  - 负责单场战斗
- `simulateBattleBatchSummary`
  - 负责同一配置下的多场统计
- `simulateBattleSensitivity`
  - 负责对多个取值点反复调用 `simulateBattleBatchSummary`

### 推荐实现位置

- 前端本地分析：
  - `src/simulator/simulateBattleSensitivity.ts`
- 后端分析：
  - `backend/simulator.py` 中新增敏感性分析入口

### 推荐执行流程

对每个扫描点执行：

1. 基于原始 `BattleInput` 复制一份输入
2. 读取当前目标属性的基准值，按 `结果值 = 基准值 + 增幅值` 计算当前扫描点的实际结果值
3. 将指定单位的指定属性替换成当前扫描点的实际结果值
4. 调用 `simulateBattleBatchSummary`
5. 收集该取值点的摘要结果，同时保留：
   - 增幅值 `value`
   - 实际结果值 `actualValue`
6. 汇总成最终分析结果

### 为什么不直接修改原 draft

因为敏感性分析本质上是“读取当前配置，生成多个变体”，而不是“改写当前编辑中的配置”。

运行期间直接修改当前 draft 会导致：

- UI 状态和结果状态耦合
- 中断后无法恢复
- 容易污染用户还没确认的配置

因此运行时必须对输入做克隆，再做局部替换。

## 校验规则

一期新增独立校验逻辑：

- `validateBattleSensitivityRequest`

### 需要校验的内容

- `input` 本身必须是合法战斗输入
- `axis.scope` 当前只能是 `unitStat`
- `unitId` 必须存在
- `field` 必须是支持扫描的单位属性字段
- `start` / `end` / `step` 必须是合法数值
- `step > 0`
- `start` / `end` 允许为负数
- `start` / `end` / `step` 必须与该属性步进对齐
- `基准值 + start` / `基准值 + end` 不得低于该属性最小值
- `(abs(end - start) / step)` 必须是整数
- `battlesPerPoint` 必须是正整数
- `pointCount` 不得超过上限
- `pointCount * battlesPerPoint` 不得超过总模拟量上限

### 一期推荐上限

- 最大扫描点数：`100`
- 最大总模拟次数：`100000`

这两个上限前后端都要同步校验。

## 运行控制

敏感性分析视图必须支持：

- 运行中禁止编辑
- 运行中可停止
- 超时自动停止

### 本地运行

本地运行时需要真正支持中途停止：

- 外层按取值点循环时检查 `AbortSignal`
- 内层批量统计时继续复用已有可取消逻辑

### 后端运行

一期后端仍沿用当前同步请求模式：

- `POST /simulate-sensitivity`

前端可以取消 `fetch` 等待，但当前 Python HTTP 服务内部计算不会真正停止。

这个限制要在文档中明确，不做误导。

## 当前推荐的一期验收标准

1. 可以选择任意一个现有单位和任意一个单位数值属性作为扫描目标。
2. 可以配置起始增幅、结束增幅、步进和每点模拟场次。
3. 本地和后端两种运行模式都能返回一致结构的分析结果。
4. 页面可以展示趋势图和明细表。
5. 趋势图图例可以控制曲线显隐，悬停时只展示当前开启曲线的数值。
6. 点击某个取值点，可以把该值应用回当前配置并切回单场模拟继续查看。
7. 非法输入会给出明确提示，不会静默纠正。
8. 总模拟量过大时会被明确拦截。
