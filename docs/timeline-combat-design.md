# 时间事件驱动改造设计与落地记录

## 文档目的

这份文档用于记录本轮战斗系统从“回合驱动”升级到“时间 + 事件驱动”的需求边界、结构设计与当前落地状态，避免后续继续迭代 `射速 / 换弹 / 弹匣` 时出现口径漂移。

## 本次已确认需求

### 业务目标

给每个单位新增 3 个属性：

- `射速（发/min）`
- `换弹动作时间（ms）`
- `弹匣最大容量（发）`

并让这 3 个属性真实参与战斗结算。

### 规则目标

- 开局弹匣默认装满。
- 每次普通攻击消耗 `1` 发弹药。
- `射速（发/min）` 决定两次开火之间的时间间隔。
- 射击间隔公式暂定为：`60000 / 射速` 毫秒。
- 弹匣打空后，单位进入换弹状态。
- 换弹持续 `换弹动作时间（ms）`。
- 换弹完成后，弹匣补满，再进入下一次可行动状态。

### 模式目标

当前仍保留两种行动结算模式：

- `Arpg即时制（时间）`
- `回合制速度高者先手`

但新增 `射速 / 换弹 / 弹匣` 后，战斗驱动不再适合继续只靠“每回合每单位一次行动”来表达，而要升级为“时间 + 事件”驱动。

## 本次非目标

- 本期不引入行动条。
- 本期不把系统改造成技能编辑器或完整技能树。
- 本期不新增复杂武器系统，例如多发齐射、预热射击、分段装填。
- 本期不修改现有伤害公式乘区口径，只改变“何时出手、能否出手、是否要换弹”这一层。

## 为什么必须从回合驱动升级

在当前实现里，单位的出手机会由“回合”驱动：

- 即时模式：本轮开始时活着，就拿到本轮出手机会
- 回合制模式：每回合按速度排顺序，依次行动

这个模型在只有“每轮 1 次普通攻击”时是够用的，但加入 `射速 / 换弹 / 弹匣` 后会立刻出现结构性问题：

1. `射速` 本质上描述的是“单位多久能再开一枪”，这是时间概念，不是回合概念。
2. `换弹动作时间` 也是持续时间，不能自然地映射到“跳过若干回合”。
3. `弹匣最大容量` 会让单位出现“连续多次开火后进入一个较长空档期”的节奏，这同样属于时间轴问题。
4. 以后如果继续加“起手前摇、技能吟唱、持续伤害、延迟爆炸、召唤物落地”等机制，都会进一步依赖统一时间轴。

因此，这次改造不应该继续在现有主循环里堆更大的 `if / else`，而应该把“伤害怎么算”和“什么时候轮到它行动”拆开。

## 结构设计结论

采用：

- `共享战斗内核 + 驱动策略 + 少量模式适配器`

不采用：

- “两套模拟器各写各的”
- “一个大基类 + 多个派生模拟器”的继承结构

### 为什么不写成两套模拟器

如果分别维护：

- 一套回合版模拟器
- 一套时间轴版模拟器

那么以下逻辑会被复制：

- 目标选择
- 命中判定
- 伤害公式
- 护甲与元素关系
- 事件日志
- 死亡与终局判定

这会直接违反 DRY。后续每次调伤害公式，都要改两遍，还很容易前后漂移。

### 为什么不优先用继承

如果做成：

- `BaseSimulator`
- `TurnBasedSimulator extends BaseSimulator`
- `TimelineSimulator extends BaseSimulator`

看起来像“抽象”，但实际风险很高：

1. 父类通常会逐渐膨胀成超大类，违反单一职责。
2. 子类会不断重写父类钩子，最后只剩下“勉强复用一点点”。
3. 后续再加第三种驱动时，继承链会越来越难维护。
4. 前后端需要同步实现同一套结构时，组合式设计比继承式设计更容易保持一致。

所以这里更适合“组合 + 策略”。

## 推荐分层

### 1. 领域协议层

职责：

- 定义输入输出类型
- 定义枚举、默认值、宏定义
- 约束哪些字段是统一宏，哪些字段是枚举

本次改造已经涉及：

- `src/config/attribute-macros.json`
- `src/domain/battle.ts`
- `src/domain/validation.ts`
- `backend/models.py`
- `backend/validation.py`

这里要把 `射速 / 换弹动作时间 / 弹匣最大容量` 纳入统一属性宏，而不是散落在 UI 或模拟器里硬编码。

### 2. 运行时状态层

职责：

- 保存单位在一场战斗里的“动态状态”
- 区分“配置输入”和“运行时变化”

建议新增或显式整理的运行时字段包括：

- `currentHp`
- `isAlive`
- `currentAmmo`
- `nextActionTime`
- `reloadUntil`
- `initialOrder`

必要时还可以继续扩展：

- `lastActionTime`
- `pendingEvents`
- `defeatedAtTime`

这一层不负责伤害公式，只负责描述“战场当前状态”。

### 3. 共享结算内核

职责：

- 只做与“单次出手如何结算”有关的事情
- 不关心外层是回合驱动还是时间驱动

建议抽成独立函数的能力包括：

- `pickTarget`
- `resolveAttack`
- `applyDamage`
- `consumeAmmo`
- `startReload`
- `finishReload`
- `evaluateBattleEnd`

这层的核心原则：

- 同一份命中规则，只实现一次
- 同一份伤害公式，只实现一次
- 同一份事件结构，只实现一次

### 4. 驱动策略层

职责：

- 决定“谁在什么时候获得行动资格”
- 决定“同一时刻的动作如何组织与提交”

建议拆成两个驱动策略：

- `TurnBasedSpeedDriver`
- `TimelineSimultaneousDriver`

#### TurnBasedSpeedDriver

语义：

- 保留现有“回合制速度高者先手”的用户认知。
- 当多个单位在同一行动时刻可出手时，按速度排序。
- `速度` 决定同一时刻谁先结算。
- `射速` 决定单位多久再次获得出手机会。

也就是说，加入射速后，这个模式不再等于“传统固定一回合一人一次行动”，而是：

- 仍然保留速度先后手语义
- 但行动频率改由射速控制

#### TimelineSimultaneousDriver

语义：

- 战斗沿统一时间轴推进。
- 每次取出当前最早的可执行时间点。
- 同一时刻开始前，先基于该时刻的快照选目标。
- 同一时刻的全部攻击先完成计算，再统一提交伤害。
- 因此允许出现同一时刻双方同时死亡。

这也是当前 `Arpg即时制（时间）` 模式后续更准确的实现方向。

## 推荐数据流

### 初始化阶段

1. 读取配置输入。
2. 生成运行时单位状态。
3. 计算初始弹匣量：
   - `currentAmmo = magazineCapacity`
4. 计算初始可行动时间：
   - 首版建议所有单位从 `time = 0` 开始可行动。

### 主循环阶段

1. 找到下一个需要处理的时间点。
2. 收集该时间点所有可行动单位。
3. 根据驱动策略决定处理顺序或同批提交方式。
4. 对每个行动单位：
   - 检查是否存活
   - 检查是否在换弹
   - 检查弹药是否足够
   - 选择目标
   - 结算攻击
   - 扣除弹药
   - 安排下一次开火或换弹结束时间
5. 统一应用该时间点的伤害结果。
6. 检查终局条件。

### 结束阶段

- 一方全灭则结束
- 即时模式下若同一时刻双方同时全灭，则记为平局
- 到达最大回合数或最大战斗时长后强制结束；若因上限结束且双方仍存活，则按平局处理

## 宏定义要求

新增属性前，必须先更新 `src/config/attribute-macros.json`，当前已加入：

- `fireRate`
- `reloadTimeMs`
- `magazineCapacity`

原因：

- 前端显示标签
- 默认值
- 最小值与步进
- 导入兼容补全
- 前后端校验
- 审计脚本

都应复用同一份定义，禁止各处重复写死。

## 已确认补充项

### 最大战斗时长

在时间轴驱动下，`maxRounds` 只适合约束“最多处理多少个时间结算点”，但不适合单独表达“整场战斗最多持续多久”。

这项现在已经确认加入：

- `最大战斗时长（ms）`

当前采用：

- 与现有 `maxRounds` 并存
- 作为额外保险阈值
- 两个条件任一先到，就结束战斗

### 首次出手时间

当前建议是：

- 所有单位开局 `time = 0` 都可行动一次

但未来也可以扩展为：

- 开场受武器前摇影响
- 首次出手受技能准备时间影响

本期先不做额外预留实现，只在结构上避免阻塞后续扩展。

## 验收标准

1. 三个新属性在前端可编辑、可导入导出、可校验。
2. 前端本地模拟与后端 API 模拟结果保持一致。
3. 即时模式下可正确处理同一时刻的同时攻击与同时死亡。
4. 回合制速度模式下，`速度` 只影响同一时刻的先后手，`射速` 影响行动频率。
5. 伤害公式与既有边界处理不因驱动改造而漂移。
